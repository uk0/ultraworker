## 2024-Q1 (1월~3월): 초기 아이디에이션 및 방향 설정

### 주요 논의 사항
- **채널 개설 목적**: 스톰 범용 플랫폼 기획을 위한 채널 (hhhssw 개설)
- **초기 컨셉**: "머리/몸/다리만 갈아끼울 수 있는 플랫폼" -> 궁극적으로 "전부 커스텀 가능한 플랫폼"으로 나아가기로 결정
- **Answer API 리스팅**: 김혜원이 answer쪽 API 리스팅 시작
- **플랫폼 화면 와꾸 작업**: 김혜원이 한시형, hhhssw에게 대략적인 화면 구성 요청
- **컨셉 문구 드래프트**: hhhssw가 스톰 플랫폼 컨셉 문구 드래프트 작성 (Google Sheets 공유)

### 주요 결정
- 스톰 플랫폼의 시작은 모듈형(머리/몸/다리 교체), 최종 방향은 완전 커스텀
- 플랫폼 미팅 시작 (김혜원, 한시형, hhhssw)

## 2024-Q2 (4월~6월): 플랫폼 설계 및 Flowise/Langflow 리서치

### 주요 논의 사항
- **VESSL AI 참고**: 김혜원이 MLOps 플랫폼 VESSL AI 공유
- **클라우드 환경 설정**: AWS/GCP 선택, 프로세서, 스토리지 등 플랫폼 설정 가능 값 리스팅
- **Figma 기획**: 스톰 플랫폼 화면 기획 시작 (Figma 링크 공유)
- **올인원 포기 결정**: 범용 플랫폼 포기, Answer 안에서만 자유도 제공 (인프라, 서비스 환경, 프롬프트 등)
- **개발 우선순위**: 김혜원이 플랫폼 기능 전체 리스팅 및 우선순위 매기기 완료 (Google Sheets 공유)
- **이미지 생성/변환 도구**: 정세민 설명을 바탕으로 이미지 생성/변환 일반화 도구 초벌 디자인
- **Flowise 리서치**: 김혜원이 Flowise 리서치 공유 -> Storm + Flowise(알맹이) 결합 방안 논의
- **Flowise 연동 고려사항**: 문서 위치 연동, 프롬프트 위치, deploy 기능, 미터링 기능
- **Langflow vs Flowise**: Langflow 운영안정성 문제로 Flowise 선택
- **dify.ai 리서치**: 김혜원이 dify.ai 리서치 -> 라이트한 버전의 플랫폼 방향
- **Pylon 분리 계획**: Open LLM으로 교체(1차) -> 프롬프팅 분리(2차) -> RAG만 남김(3차) -> 오케스트레이션 개발 순서

### 주요 결정
- 올인원 플랫폼 포기, Answer 내에서 자유도 제공 방향
- Langflow 대신 Flowise 채택 (운영안정성)
- 마크스폰, 한국자산평가 케이스로 Flowise 검증
- Carrier 모듈 분리 결정 (명하님 선장)

### 주요 참여자
- 김혜원(기획/PM), 한시형(사업/방향), 정세민(프롬프트엔지니어링), 송명하(인프라), kozistr833(엔진)

## 2024-Q3 (7월~9월): Carrier MVP 개발 및 연동

### 주요 논의 사항
- **모듈 역할 분담**:
  - Stargate: 계정/팀/권한, 로깅, 빌링, 주문서 만들기, deploy
  - Carrier: 스타게이트에서 주문서 받아서 구동 (별도 분리 확정)
  - 문서 업로드 모듈: 파일론에서 분리 필요
- **Carrier MVP 테크스펙**: 김덕현(Dekay)이 Carrier MVP 테크스펙 작성 및 Notion 공유
- **주문서(Order Sheet) 포맷**: YAML 형태로 작성, 여러 차례 수정/개선
- **Carrier 개발**: swkim0512(수원)이 Carrier 개발, 코틀린/스프링 베이스라인
- **노드 시스템**: storm/llm, storm/rag 노드 타입 정의, import/export properties 설계
- **API 연동**: Carrier <-> Stargate <-> Open LLM <-> RAG 연동 작업
- **QA 진행**: 김혜원이 Carrier QA 진행, 다수 이슈 발견 및 수정
  - 에러 코드 정의 (RAG, Open LLM)
  - 로그 파일 저장 및 조회
  - Task ID + Node ID 헤더 추가
  - 간헐적 커넥션 끊김 이슈
  - Stream 처리 이슈 (SSE vs callback 방식 논의)
- **디폴트 주문서**: 기본 주문서 작성 완료
- **추가 기능 요청**: 프롬프트에 API로 받아온 변수값 삽입 기능 (띵스플로우 요청)
- **라이브 배포**: 스게 라이브 배포 완료 (2024-09-12)

### 주요 결정
- Carrier 코틀린/스프링으로 개발
- 주문서 포맷: YAML/JSON, 템플릿 변수 {{변수명}} 형식
- Stream 처리: 마지막 노드만 스트림, sequence_body에 스트림 여부 추가
- Pylon v2 경로 변경 대응
- Carrier 라이브 배포 준비

### 주요 이슈
- 스트림 응답 관련 캐리어-스게 연동 이슈 (callback vs event-stream)
- 에러 처리 표준화 필요
- contexts API 일반화 보류
